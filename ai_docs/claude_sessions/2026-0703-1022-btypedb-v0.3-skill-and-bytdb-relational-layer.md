## Part 2: bytdb (repo: `~/projs/go/bytdb`, github.com/rohanthewiz/bytdb)

Answered "what would it take to build a relational layer on btypedb like CRDB uses Pebble", then built the first two milestones. Design stance: one `btypedb.DB[string, []byte]` keyspace; single-writer serializable (SQLite niche, no MVCC); btypedb's comparator indexes deliberately unused — SQL indexes are encoded key ranges (persistent for free).

### Milestone 1 (`326f560`)
- **`tuple` package**: order-preserving encoding; type tags NULL(0x01) < false < true < int(0x04) < float < bytes < string(0x07) — tags are persistent format, never renumber. Sign-flipped big-endian int64; float bit-transform (`-0.0` normalized to `+0.0`, NaN rejected); 0x00 escaped as `0x00 0xFF` with `0x00 0x01` terminator so `"a" < "a\x00" < "ab"`; `PrefixEnd` for scan bounds. Property test: 400 random tuples (byte gen skewed to 0x00/0x01/0xFF), all pairs `bytes.Compare(enc)` ≡ `Compare(tuples)`; pinned tricky-order table; corrupt-decode tests. Gotcha fixed: empty `[]byte` must decode non-nil.
- **Table layer**: keys `tuple(tableID, 1, pk...) → tuple(non-pk cols)`; catalog = JSON descriptors as rows of system table ID 1 keyed by name; table-ID sequence at key `tuple(0)` (starts at 100), allocated atomically with the descriptor write. `CreateTable`/`DropTable`, `Insert` (type coercion, NULL-in-PK rejected, PK uniqueness in-tx), `Get`/`Delete`, `Scan`/`ScanRange` in PK order with **partial-prefix bounds** on composite keys.

### Milestone 2 (`95f95dd`) — secondary indexes
- Two entry forms, told apart by tuple arity on decode:
  - non-unique: `tuple(tid, iid, indexed..., pk...) → ()`
  - unique: `tuple(tid, iid, indexed...) → tuple(pk...)` — uniqueness by key collision; rows with NULL in any indexed column use the pk-suffixed form even in unique indexes (NULLs never conflict, SQL/CRDB semantics).
- `CreateIndex(table, name, unique, cols...)`: backfill (collect-then-write inside one `Update` — avoids mutating while iterating the tx), uniqueness check, descriptor write — all one atomic commit; failed backfill leaves no trace. `DropIndex`/`DropTable` `DeleteRange` their spaces (DropTable now clears `tuple(tableID)` whole-space, not just the primary range — was a latent m1 bug once indexes existed).
- **Schema-visibility design**: DML resolves its descriptor *inside* its own kv `Update`; DDL publishes the new descriptor (under `e.mu`) as the *last step inside* its kv `Update`; `ddlMu` serializes DDL. Since btypedb serializes writable txs, an insert serialized after CreateIndex always sees the index. Race-tested (`TestConcurrentInsertsDuringCreateIndex`, run 3× under `-race`).
- `ScanIndex(table, index, from, to)` runs on a `Begin(false)` snapshot — **must not** nest `kv.Get` inside a DB-level `Ascend` loop (recursive RLock can deadlock with a queued writer); the tx makes entry→row resolution lock-free and consistent.
- Insert/Delete maintain all indexes in the row's commit; Delete reads+decodes the old row to compute entry keys.
