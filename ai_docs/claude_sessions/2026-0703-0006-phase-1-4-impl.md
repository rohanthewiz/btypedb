## Origin

Started from the question: *"Tidwall has this data structure called btype — what would it take to make a pure-Go embedded database out of it?"* Key insight from research: btype's B-tree collections have **O(1) reference-counted copy-on-write snapshots**, which buy the three normally-hard database features — lock-free snapshot-isolated readers, cheap transactions (copy → mutate → pointer swap), and non-blocking persistence. Constraint accepted up front: btype is memory-only, so this is a memory-resident DB with disk durability (BuntDB/Redis model), not a disk-pager like bbolt. Dataset must fit in RAM.

## Architecture

```
btypedb/
  db.go       DB[K cmp.Ordered, V any], Open/replay, direct ops, options, syncer
  state.go    dbState: data Map + ttl Map + exp Table + idx map — swapped as one pointer
  tx.go       Tx: Begin/View/Update/Commit/Rollback over COW state snapshots
  wal.go      record framing + streaming replay with torn-tail recovery
  compact.go  Compact() + auto-compaction (snapshot rewrite, atomic rename)
  ttl.go      background expiry sweeper
  index.go    CreateIndex/DropIndex + index iteration (DB and Tx)
  codec.go    Codec[T] struct-of-funcs; String/Bytes/Int64/Uint64/JSON codecs
```

### WAL format
Single append-only file of CRC-framed records:
`op(1) | klen(4) | vlen(4) | key | val | crc32(4)` (little-endian).
Ops: `set(1)`, `delete(2)`, `batch(3)` (val = uint64 count; next N records apply all-or-nothing), `setttl(4)` (val prefixed with 8-byte absolute deadline, unix nanos). Replay stops at the first torn/CRC-failing record and truncates there; a torn batch is discarded whole (transaction atomicity across crashes).

### Concurrency model (the load-bearing decisions)
- Verified in btype source: refcounts are **atomic** (`atomic.AddInt64(&n.rc, ...)`); shared nodes are copied, never mutated in place. But `Copy()` mutates the *source's* bookkeeping (`copied` flag, root rc), so **all Copy/Release/root-swap ops happen under `db.mu`**; snapshot traversal is lock-free.
- Lock order: `writerMu` → `mu`, always. `writerMu` serializes write txs + direct writes (single writer). `compactMu` serializes compactions; never held with `writerMu`.
- Commit: append batch to WAL (1 write + 1 fsync) → swap `db.state` pointer → release old state.
- Known footgun (documented): calling `db.Set` inside `Update` self-deadlocks on `writerMu`; use Tx methods.

### dbState refactor (phase 4 enabler)
`db.m` → `*dbState` holding data map, ttl map (key→deadline), exp table ((deadline,key) ordered, custom comparator), and idx map. Tx copies the whole struct (each tree O(1)); commit swaps one pointer → data/TTLs/indexes always change atomically together. Used `Table` not `Prique` for expiry: exact-entry `Delete` + `Front()` beats a largest-first heap for TTL replacement.

### Compaction protocol
Phase A (brief lock): O(1) state snapshot + record log offset. Stream (no locks): write snapshot as set/setttl records to `path.compact`, dropping expired keys. Phase B (brief lock): splice tail written since snapshot via `SectionReader`, fsync, atomic `rename()`, swap file handle, best-effort dir fsync. Crash-safe: temp only becomes live via rename; stale `.compact` discarded on Open. Auto-trigger (BuntDB heuristic): log ≥ minSize (32MB default) and grown growthPct (100% default) past post-compaction size.

### TTL
Lazy expiry at read time (Get/Contains/All/index iteration skip expired) + background sweeper (default 500ms, `WithSweepInterval`, ≤0 disables) that deletes expired keys as ordinary batched delete txs — candidates scanned under RLock, deadlines rechecked inside the tx. Absolute deadlines survive replay/compaction. Plain `Set` clears TTL. `Len()` counts physical entries (may include expired-unswept) — documented.

### Secondary indexes
BuntDB-style: an index = comparator over (K,V) pairs (primary key tiebreak), not a separate key type — keeps generics workable. Backfilled at CreateIndex (O(n log n), writers paused), maintained by every write, transactional (tx sees own uncommitted writes in index order; rollback discards). Not persisted (comparators aren't serializable) — re-register after Open. `writerMu` in CreateIndex prevents interleaving with open write txs (whose commit swaps the whole state pointer).

### DeleteRange
`[min, max)`, on DB (via Update) and Tx. Encodes all keys *before* mutating (codec failure leaves tx intact), logs as one atomic batch, returns count of *visible* deletions while also scrubbing expired-unswept keys from all trees.

## Testing
41 tests, all green under `-race` (multiple consecutive runs):
- CRUD, reopen/replay, torn-tail + CRC-corruption recovery
- Tx visibility/rollback/snapshot isolation/batch replay/torn-batch atomicity
- Compaction: shrink+preserve, racing concurrent writes, auto-trigger, stale temp cleanup
- **SIGKILL crash suite** (`crash_test.go`): test binary re-execs itself as a write-hammering child (8-key txs, direct sets, deletes, 8KB compaction threshold), parent kills at varied offsets ×6 rounds, verifies every batch group is all-or-nothing with uniform values. Caveat: SIGKILL doesn't lose page cache → not power-loss-grade.
- TTL: expiry visibility, persistence across reopen, sweeper physical removal, Set-clears-TTL, tx TTL, compaction drops expired/preserves deadlines
- Indexes: backfill, overwrite-moves-never-duplicates, transactionality, expiry skipping, drop/unknown
- Concurrency hammer: 2 tx writers + direct writer + sweeper + TTL writes vs 4 snapshot readers; assertions kept time-monotonic (index iterated later can only see *fewer* live keys) to avoid wall-clock flakes

## Key API decisions
- `Codec[T]` is a struct of Encode/Decode funcs (not interface) so Go infers K/V at `Open(path, StringCodec, JSONCodec[User]())`. Codecs are disk-only; in-memory ordering is K's natural `cmp.Ordered` order.
- Failed WAL append → sticky `writeErr`, DB goes read-only (writes after a torn record would be stranded past the recovery truncation point).
- Delete of absent key logs nothing; delete of expired-unswept key physically removes but reports `existed=false`.
- Sync policies: `SyncAlways` (default) / `SyncEverySecond` / `SyncNever`.
- Errors: serr-wrapped throughout; sentinels (`ErrClosed`, `ErrTxClosed`, `ErrTxNotWritable`) are plain `errors.New` so `errors.Is` works through serr chains.
